# 18. SEO Content System

> 본 문서는 검색 유입을 위한 콘텐츠 시스템을 정의한다.
> 360개 오름 백과 페이지의 표준 양식, 메타 데이터, 구조화 데이터, 콘텐츠 제작 워크플로우를 다룬다.
> 본 문서는 출시 전 가장 빨리 작업이 시작되어야 하는 영역이다 (콘텐츠 제작 시간 + 검색 엔진 인덱싱 시간).

---

## 0. 전략 개요

### 0.1 목표

**"제주 오름"** 관련 검색에서 우리 사이트가 최상위에 노출되어, 검색 유입 → 콘텐츠 탐색 → 100선 인지 → 패스포트 구매로 이어지는 전환 깔때기를 만든다.

### 0.2 핵심 가설

1. 제주 오름에 대한 **체계적·전문적 단일 소스가 부재**한 시장 — 우리가 그 자리를 차지할 수 있다
2. 360개 오름 모두 SEO 페이지를 가지면 **롱테일 키워드를 광범위하게 점유**할 수 있다
3. 100선 외 페이지는 **트래픽 깔때기 역할**을 하며 100선 페이지로 사용자를 유도한다
4. 사용자 후기·사진이 자동으로 노출되는 **신선한 콘텐츠**는 SEO 점수를 지속적으로 끌어올린다

### 0.3 KPI (페이즈별)

| 시점 | 핵심 KPI |
|------|---------|
| 출시 시점 | SEO 페이지 150~200개 인덱싱 완료 |
| 출시 +3개월 | 월 검색 유입 1만 명, "제주 오름" 검색 결과 1페이지 진입 |
| 출시 +6개월 | 월 검색 유입 5만 명, 360개 페이지 완성 |
| 출시 +12개월 | 월 검색 유입 20만 명, "제주 오름" 검색 결과 1위 |

## 1. 정보 아키텍처

### 1.1 3계층 구조

```
1계층: 허브 페이지 (1개)
├─ /oreum  ← "제주 오름이란"
│
2계층: 카테고리 페이지 (10~15개)
├─ /oreum/region/east        ← "제주 동부 오름"
├─ /oreum/region/west        ← "제주 서부 오름"
├─ /oreum/region/south       ← "제주 남부 오름"
├─ /oreum/region/north       ← "제주 북부 오름"
├─ /oreum/region/central     ← "제주 중산간 오름"
├─ /oreum/level/beginner     ← "초보자 오름"
├─ /oreum/level/explorer     ← "익스플로러 오름"
├─ /oreum/season/spring      ← "봄 오름"
├─ /oreum/season/summer      ← "여름 오름"
├─ /oreum/season/autumn      ← "가을 오름"
├─ /oreum/season/winter      ← "겨울 오름"
├─ /oreum/feature/sunrise    ← "일출 명소 오름"
├─ /oreum/feature/sunset     ← "일몰 명소 오름"
└─ /oreum/feature/crater     ← "분화구 오름"
│
3계층: 개별 오름 페이지 (360개)
├─ /oreum/saebyeol           ← 새별오름
├─ /oreum/darangshi          ← 다랑쉬오름
├─ /oreum/yongnuni           ← 용눈이오름
└─ ... (총 360개)
```

### 1.2 URL 규칙

**원칙**: 짧고, 의미 있고, 영구적이다.

| 페이지 | URL 패턴 |
|--------|---------|
| 허브 | `/oreum` |
| 카테고리 | `/oreum/{type}/{slug}` |
| 개별 오름 | `/oreum/{slug}` |
| 사용자 후기 | `/oreum/{slug}/reviews` |
| 갤러리 | `/oreum/{slug}/gallery` |

**slug 규칙**:
- 로마자 표기 (한국·글로벌 검색 둘 다 잡기 위해)
- 모두 소문자, 공백 대신 하이픈
- 예: `saebyeol` (새별오름), `darangshi` (다랑쉬오름)
- 한국식 표기 우선 (예: `darangshwi`보다 `darangshi`가 더 일반적)

대안: 한글 slug (`/oreum/새별오름`)도 검토했지만 URL 인코딩 문제 + 글로벌 노출 약화로 미채택.

### 1.3 내부 링크 구조

각 페이지가 다른 페이지와 어떻게 연결되는지가 SEO의 핵심.

**개별 오름 페이지에서 자동 생성되는 내부 링크**:
- 같은 region 오름 5~10개 (`oreum_links` 테이블의 `same_region`)
- 비슷한 difficulty 오름 3~5개 (`similar_difficulty`)
- 추천 코스에 함께 묶이는 오름 2~3개 (`recommended_combo`)
- 가까운 오름 5개 (`nearby`)
- 카테고리 허브 페이지 (해당 region/level/season)
- 1계층 허브 페이지

**카테고리 페이지에서**:
- 해당 카테고리의 모든 오름 (3계층)
- 다른 카테고리 (2계층)
- 1계층 허브

**허브 페이지에서**:
- 모든 카테고리 (2계층)
- 인기 오름 Top 10 (3계층)

이 구조로 각 페이지가 평균 **10~20개 내부 링크**를 갖게 되어 사이트 전체의 권위가 분산·강화된다.

## 2. 표준 페이지 양식

### 2.1 개별 오름 페이지 (3계층) — 핵심 양식

모든 360개 오름 페이지가 동일한 구조를 갖는다. 이 일관성이 SEO에 결정적이다.

#### 2.1.1 페이지 구조

```
┌──────────────────────────────────────┐
│ <head> (메타 태그, JSON-LD)             │
├──────────────────────────────────────┤
│ Header (사이트 공통)                    │
├──────────────────────────────────────┤
│ Breadcrumb                          │
│ 홈 > 제주 오름 > 동부 > 다랑쉬오름        │
├──────────────────────────────────────┤
│ 상단 카드 영역                         │
│ - 카드 일러스트 (100선) 또는 대표 사진    │
│ - 한 줄 요약                          │
│ - 핵심 메타 (표고/지역/난이도)           │
├──────────────────────────────────────┤
│ 본문 (10개 표준 섹션)                   │
├──────────────────────────────────────┤
│ CTA 영역                              │
├──────────────────────────────────────┤
│ 관련 오름 (내부 링크 영역)               │
├──────────────────────────────────────┤
│ Footer                               │
└──────────────────────────────────────┘
```

#### 2.1.2 본문 10개 표준 섹션

각 섹션은 H2 또는 H3 헤딩을 가지며, 모든 오름 페이지가 동일한 순서·헤딩명을 사용한다.

**섹션 1: 한눈에 보기** (H2)
- 좌: 핵심 정보 카드 (표고, 비고, 난이도, 코스 거리, 소요 시간)
- 우: 위치 지도 (정적 이미지 또는 임베드)
- 분량: 100~150자 본문 + 표

**섹션 2: 오름 소개** (H2)
- 이름의 유래 (한자, 옛 지명, 전설 등)
- 형성 배경 (언제·어떻게 만들어졌는지)
- 역사·문화적 의미 (목축, 신앙, 마을과의 관계)
- 분량: 250~350자

**섹션 3: 지질·생태 정보** (H2)
- 화산학적 분류 (분석구, 응회환 등)
- 지질학적 특징
- 식생, 동물상
- 보호 가치 (있다면)
- 분량: 200~300자

**섹션 4: 탐방 정보** (H2)
- 탐방 코스 설명
- 거리, 소요 시간, 누적 표고차
- 노면 상태 (흙길/돌계단/데크 비율)
- 난이도 분석
- 권장 장비
- 분량: 250~400자

**섹션 5: 가는 길** (H2)
- 자가용 안내 (주소, 주차장 위치, 수용 대수, 무료/유료)
- 대중교통 (버스 노선, 정류장에서 도보 시간)
- 운영 시간 (있다면)
- 입장료 (있다면)
- 분량: 150~200자

**섹션 6: 베스트 시즌과 시간대** (H2)
- 봄/여름/가을/겨울 각 계절의 특징
- 추천 시즌과 비추천 시즌
- 일출/일몰 시간 (계절별)
- 시간대별 추천 (이른 아침/오후/저녁)
- 분량: 200~300자

**섹션 7: 포토 포인트와 볼거리** (H2)
- 인기 포토 스팟 위치
- 정상에서 보이는 풍경
- 특별한 볼거리 (분화구, 능선, 식생 등)
- 사진 갤러리 (사용자 기여 사진 자동 노출)
- 분량: 200~250자

**섹션 8: 주변 함께 가볼 곳** (H2)
- 인근 오름 (3~5개, 거리·도보 시간 표시)
- 인근 제휴 상권 (있다면)
- 인근 맛집·카페 (제휴 외)
- 함께 묶어 가기 좋은 코스 추천
- 분량: 150~200자

**섹션 9: 방문자 후기** (H2)
- 사용자 코멘트 자동 노출 (최신 5~10개)
- 평점 분포 (있다면)
- "이 오름에 다녀온 후기 남기기" CTA
- 분량: 동적 (사용자 콘텐츠)

**섹션 10: 관련 오름** (H2)
- 자동 생성된 내부 링크 (`oreum_links` 활용)
- 카테고리: "같은 동부 오름", "비슷한 난이도", "추천 코스"
- 각 5~10개 카드 형태로

#### 2.1.3 분량 가이드

총 분량: **1500~2500자** (한글 기준, 공백 제외)

이 분량이 중요한 이유:
- 1500자 이하: thin content로 SEO 페널티 위험
- 2500자 초과: 사용자 이탈, 핵심 정보 희석

## 3. 메타 데이터 표준

### 3.1 메타 태그

각 페이지에 필수로 포함되는 메타 태그.

```html
<!-- 기본 메타 -->
<title>{제목} | 제주 오름 패스포트</title>
<meta name="description" content="{155자 이내 설명}">
<meta name="keywords" content="{쉼표 구분 키워드}">
<link rel="canonical" href="https://jejuoreum.com{경로}">

<!-- Open Graph (페이스북, 카카오톡 공유 시) -->
<meta property="og:title" content="{제목}">
<meta property="og:description" content="{설명}">
<meta property="og:image" content="{이미지 URL, 1200x630}">
<meta property="og:url" content="{전체 URL}">
<meta property="og:type" content="article">
<meta property="og:site_name" content="제주 오름 패스포트">
<meta property="og:locale" content="ko_KR">

<!-- Twitter Card -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="{제목}">
<meta name="twitter:description" content="{설명}">
<meta name="twitter:image" content="{이미지 URL}">

<!-- 모바일 최적화 -->
<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=5">

<!-- 검색엔진 -->
<meta name="robots" content="index, follow">
```

### 3.2 Title 작성 규칙

**개별 오름 페이지**:
- 패턴: `{오름명} - {핵심 특징} | 제주 오름 100선 | 제주 오름 패스포트`
- 예: `다랑쉬오름 - 오름의 여왕, 표고 382m | 제주 오름 100선 | 제주 오름 패스포트`
- 60자 이내 (한글 기준)
- 핵심 키워드를 앞쪽에 배치

**카테고리 페이지**:
- 패턴: `{카테고리} - {부제} | 제주 오름 패스포트`
- 예: `제주 동부 오름 30선 - 우도부터 따라비까지 | 제주 오름 패스포트`

**허브 페이지**:
- `제주 오름의 모든 것 - 360개 오름 백과 | 제주 오름 패스포트`

### 3.3 Description 작성 규칙

**개별 오름 페이지**:
- 155자 이내 (구글 SERP 표시 한계)
- 자연스러운 문장 (키워드 나열 금지)
- 매력적인 후크 + 핵심 정보 + CTA
- 예: `제주 동부의 다랑쉬오름은 표고 382m로 '오름의 여왕'으로 불립니다. 깊은 분화구와 사방의 풍경이 일품인 오름의 탐방 코스, 가는 길, 베스트 시즌을 안내합니다.`

**카테고리 페이지**:
- 카테고리 특징 + 포함된 오름 수 + CTA
- 예: `제주 동부에는 다랑쉬, 용눈이, 따라비 등 30여 개의 명품 오름이 모여 있습니다. 동부 오름의 특징, 추천 코스, 베스트 시즌을 한눈에.`

### 3.4 Keywords (메타 태그 + 콘텐츠)

각 오름 페이지가 타깃하는 키워드:

**1차 키워드** (가장 중요):
- "{오름명}" — 예: "다랑쉬오름"
- "{오름명} 등산" — "다랑쉬오름 등산"
- "{오름명} 가는 길" — "다랑쉬오름 가는 길"

**2차 키워드** (롱테일):
- "{오름명} 주차장"
- "{오름명} 코스"
- "{오름명} 일출"
- "{오름명} 추천"
- "{오름명} 시간"

**3차 키워드** (의미 연관):
- "제주 {지역} 오름"
- "제주 {난이도} 오름"
- "제주 {계절} 오름"

이 키워드들이 자연스럽게 본문에 녹아 있어야 한다 (키워드 스터핑 금지).

## 4. 구조화 데이터 (JSON-LD)

검색 결과에 rich snippet 표시되도록 구조화 데이터 추가. 클릭률 평균 30% 향상.

### 4.1 개별 오름 페이지

```json
{
```

## 5. sitemap.xml

### 5.1 자동 생성

빌드 시점 또는 nightly cron으로 자동 생성. Next.js의 `app/sitemap.ts` 활용.

```typescript
// app/sitemap.ts
export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const oreums = await getAllPublishedOreums();
  const categories = await getAllCategories();

  return [
    {
      url: 'https://jejuoreum.com',
      lastModified: new Date(),
      changeFrequency: 'weekly',
      priority: 1.0,
    },
    {
      url: 'https://jejuoreum.com/oreum',
      lastModified: new Date(),
      changeFrequency: 'weekly',
      priority: 0.9,
    },
    ...categories.map(cat => ({
      url: `https://jejuoreum.com/oreum/${cat.type}/${cat.slug}`,
      lastModified: cat.updated_at,
      changeFrequency: 'weekly',
      priority: 0.8,
    })),
    ...oreums.map(oreum => ({
      url: `https://jejuoreum.com/oreum/${oreum.slug}`,
      lastModified: oreum.updated_at,
      changeFrequency: 'monthly',
      priority: oreum.is_top_100 ? 0.9 : 0.6,
    })),
  ];
}
```

### 5.2 sitemap 분할

페이지 수가 많아지면 sitemap을 분할:
- `sitemap.xml` (인덱스)
- `sitemap-categories.xml`
- `sitemap-oreums-1.xml` (1~100)
- `sitemap-oreums-2.xml` (101~200)
- `sitemap-oreums-3.xml` (201~360)

## 6. 콘텐츠 제작 워크플로우

### 6.1 작업 분량 추정

- 360개 오름 × 평균 1800자 = **약 65만 자**
- 책으로 약 5~6권 분량
- 1인 작성 시 6개월~1년

### 6.2 3단계 워크플로우

#### 6.2.1 1단계: 자동 템플릿 생성 (가장 빠름)

기존 360개 메타 DB를 기반으로 1차 드래프트 자동 생성. Node.js 스크립트로 일괄 처리.

**자동 생성 가능 영역**:
- 섹션 1 "한눈에 보기" — 정형 데이터 그대로 (표고, 거리 등)
- 섹션 5 "가는 길" — 좌표 기반 자동 작성
- 섹션 6 "베스트 시즌" — 메타 데이터의 `recommended_seasons` 활용
- 섹션 8 "주변 함께 가볼 곳" — `oreum_links` 자동 활용
- 섹션 10 "관련 오름" — 완전 자동

**자동 생성 어려운 영역** (AI 또는 인간 필요):
- 섹션 2 "오름 소개" — 이름 유래, 역사
- 섹션 3 "지질·생태"
- 섹션 4 "탐방 정보" 의 서술 부분
- 섹션 7 "포토 포인트" 의 묘사

이 단계 결과: 한 오름당 약 600~800자 골격.

#### 6.2.2 2단계: AI 콘텐츠 보강

자동 생성된 골격에 LLM(Claude/GPT)으로 살을 붙인다.

**프롬프트 예시**:
```
다음은 제주 오름 한 곳에 대한 백과 페이지의 일부입니다.
주어진 메타 데이터를 바탕으로, 누락된 섹션을 자연스럽게 작성해주세요.

[메타 데이터]
- 이름: 다랑쉬오름
- 지역: 제주 동부, 제주시 구좌읍
- 표고: 382m, 비고: 227m
- 분류: 분석구
- 별칭: 월랑봉

[기존 자동 생성 섹션]
{섹션 1, 5, 6, 8, 10 텍스트}

[작성이 필요한 섹션]
- 섹션 2: 오름 소개 (이름의 유래, 형성 배경, 역사·문화적 의미)
- 섹션 3: 지질·생태 정보
- 섹션 4: 탐방 정보 서술
- 섹션 7: 포토 포인트와 볼거리

[작성 가이드]
- 각 섹션 200~350자
- 정중하고 정보적인 톤
- 키워드 자연스럽게 녹이기 (오름명, 지역명, 등산, 가는 길)
- 검증되지 않은 정보는 작성하지 말 것
- 사실 확인이 필요한 부분은 [확인 필요] 표시
```

**중요**:
- AI가 생성한 후 **반드시 사실 확인** 필요 (할루시네이션 위험)
- 검증되지 않은 사실은 [확인 필요] 표시 후 인간 검수 단계로 전달

이 단계 결과: 한 오름당 약 1500자.

#### 6.2.3 3단계: 인간 큐레이션 (가장 가치 있는 단계)

기획자(오름 전문가) 또는 콘텐츠 작가가 다음을 수행:

- AI 생성 부분의 **사실 확인 및 수정**
- AI가 못 쓰는 **개인적 관찰, 현장 디테일, 알려지지 않은 팁** 추가
- 톤 통일 (모든 페이지가 같은 목소리를 갖게)
- [확인 필요] 표시된 부분 처리

**시간 투자 권장**:
- 100선 60개 (트래픽 최다 예상): 한 페이지당 1~2시간 깊이 작성
- 100선 나머지 40개: 한 페이지당 30~40분
- 100선 외 260개: 한 페이지당 15~20분 (빠른 검수)

### 6.3 우선순위 전략

**페이즈 0 (출시 전)**:
- 100선 100개: 모두 3단계 완료 (인간 큐레이션 포함)
- 100선 외 50~100개: 1+2단계 완료 (AI 드래프트 + 빠른 검수)
- 카테고리 페이지 10~15개: 인간 작성
- 허브 페이지 1개: 인간 작성

**페이즈 1 (출시 후)**:
- 매주 10개씩 100선 외 페이지 추가
- 8~9개월 안에 360개 완성

**페이즈 2 이후**:
- 사용자 후기 자동 통합
- 시계열 콘텐츠 갱신 (계절 변화 등)
- 인기 페이지 심화 보강

### 6.4 콘텐츠 갱신 주기

- **필수 갱신**: 탐방로 변경, 주차장 폐쇄 등 사실 변경 시 즉시
- **분기 갱신**: 사용자 후기 통합, 사진 추가
- **반기 갱신**: 본문 업데이트, 키워드 재최적화
- **연 갱신**: 전체 톤 점검, 구조 개선

## 7. 콘텐츠 작성 가이드

### 7.1 톤앤매너

- **정중하고 정보적**: 친구 간 잡담이 아닌 전문적 가이드 톤
- **친절하고 따뜻한**: 딱딱한 백과사전이 아닌 안내자의 톤
- **간결**: 한 문장이 너무 길지 않게 (한글 50자 이내 권장)
- **단정**: "~합니다" 체로 통일 ("~해요" 체 미사용)

좋은 예: `다랑쉬오름은 표고 382m로 제주 동부에서 가장 높은 오름 중 하나입니다.`
나쁜 예: `다랑쉬오름이라는 오름이 있어요! 엄청 높아서 382m나 된다고 해요.`

### 7.2 어휘 가이드

**선호 어휘**:
- 오름 (○) vs 화산구 (△, 학술 맥락에서만)
- 탐방 (○) vs 등산 (△, 너무 격식)
- 정상 (○) vs 봉우리 (△)
- 분화구 (○) — 그대로
- 시즌 / 계절 (○) 둘 다 가능
- 베스트 / 최고 (○) 둘 다 가능

**피할 어휘**:
- "꼭 가봐야 할" 같은 과장 표현 (SEO에 불리)
- "최고의", "최강의" 같은 단정적 형용사 (남발 금지)
- 광고성 표현 ("강력 추천!", "놓치지 마세요!")
- 외국어 남발 (한글 표현 우선)

### 7.3 키워드 자연스러운 배치

**나쁜 예 (키워드 스터핑)**:
> 다랑쉬오름 등산을 위한 다랑쉬오름 가는 길과 다랑쉬오름 주차장 정보를 다랑쉬오름 코스와 함께 안내합니다.

**좋은 예**:
> 다랑쉬오름까지 가는 길은 어렵지 않습니다. 제주시에서 차로 약 50분 거리이며, 입구에 30대 규모의 무료 주차장이 마련되어 있습니다. 등산 코스는 하나로 단순하며, 정상까지 약 1.3km를 천천히 걸어 올라가면 됩니다.

후자에는 같은 키워드들이 자연스러운 문장 안에 녹아 있다.

### 7.4 사실 검증 정책

**검증 가능한 정보만 작성한다.**

- 표고, 좌표, 행정구역 → DB의 신뢰 가능 데이터 사용
- 거리, 소요 시간 → 실측 또는 공식 자료
- 역사, 유래 → 검증 가능한 출처 (제주관광공사, 학술 자료)
- 식생, 동물상 → 환경 자료

**금지**:
- "~라고 전해진다" 같은 모호한 표현
- 검증되지 않은 전설을 사실처럼 서술
- 개인 의견을 객관 사실로 포장

## 9. 한국 검색 특화

### 9.1 네이버 검색 대응

네이버는 자체 콘텐츠(블로그, 카페, 지식인)를 우선 노출하는 경향이 있어 외부 사이트 진입 장벽이 높다. 대응 전략:

**1. 콘텐츠 신선도 유지**
- 정기적 갱신 (월 1회 이상)
- 사용자 후기 자동 노출 (자동 갱신 효과)

**2. 깊이 있는 콘텐츠**
- 1500~2500자 본문이 표준
- 단순 정보가 아닌 깊이 있는 가이드

**3. 백링크 확보 전략**
- 네이버 블로그 미러링 (간략 버전 + 본문 링크)
- 제주 관련 카페 활동
- 보도자료 배포 (런칭 시점)

**4. 네이버 지도 등록**
- 각 오름 위치를 네이버 지도에 등록
- 우리 사이트 링크 연결

### 9.2 모바일 최적화

한국 사용자의 80%+ 모바일 검색.
- 모바일 페이지 속도 (3G 기준 5초 이내)
- 모바일 가독성 (글자 크기, 간격)
- AMP 도입 검토 (단, 최근 영향력 약화)

### 9.3 한글 검색 최적화

- 한글·영문 변환 가능한 키워드 모두 노출 (예: "오름", "Oreum" 둘 다)
- 자주 틀리는 표기 대응 (예: 다랑쉬오름 vs 다랑쉬 오름 vs 다랑쉬봉)
- 별칭·이명도 alt_names에 포함하여 노출

### 9.4 다국어 SEO 전략

**페이즈별 다국어 SEO 도입 계획**:

| 시점 | 대상 언어 | 범위 |
|------|----------|------|
| 페이즈 0~1 | 한국어 | 100% (UI + 콘텐츠) |
| 페이즈 2 | 영어 | UI 텍스트만 |
| 페이즈 3 | 영어 | 100선 100개 SEO 콘텐츠 |
| 페이즈 4 | 일본어 | UI + 일부 SEO |
| 페이즈 4 | 중국어 | UI + 일부 SEO |

**기술적 구현**:
- URL 구조: `/ko/oreum/{slug}` / `/en/oreum/{slug}` / `/ja/oreum/{slug}` / `/zh/oreum/{slug}`
- `hreflang` 태그로 언어별 페이지 명시
- 언어별 sitemap 분할
- 자동 언어 감지 (`Accept-Language`)

**hreflang 태그 예시**:

## 10. 부가 수익화 (페이즈 3+)

SEO 트래픽이 충분히 확보된 후 검토 가능한 수익원.

### 10.1 제휴 마케팅 (Affiliate)

각 오름 페이지에 자연스러운 위치에 노출:
- "제주 호텔 예약하기" → 호텔 예약 사이트 어필리에이트
- "제주 렌터카" → 렌터카 어필리에이트
- "제주 항공권" → 항공 어필리에이트

수수료: 평균 3~10%.

**원칙**:
- 사용자 경험을 해치지 않는 자연스러운 위치
- 클릭 유도형 광고 강요 금지
- 본문과 명확히 분리 (광고 표시)

### 10.2 지역 광고

오름 근처 카페·식당의 유료 광고:
- 카테고리 페이지 하단
- 개별 오름 페이지 "주변" 섹션

**구분**:
- 제휴 상권: 무료 노출 (이미 연 제휴료 받음)
- 일반 광고: 유료 (월 정액 또는 노출당 과금)

### 10.3 콘텐츠 라이선싱

충분히 쌓인 후:
- 출판사: 제주 가이드북 콘텐츠 라이선스
- 여행사: 360개 오름 데이터 API 라이선스
- 학술 기관: 시계열 데이터 (식생, 탐방로 변화)

### 10.4 광고 도입 시점·원칙

**도입 안 할 시기**:
- 페이즈 0~2: 사용자 신뢰 구축 우선
- 사용자 1만 명 이전

**도입 시점**: 페이즈 3 이후, 충분한 트래픽과 신뢰 확보 후

**원칙**:
- 첫 화면 광고 금지
- 본문 중간 광고 최소화
- 명확한 광고 표시
- 모바일에서 콘텐츠 가리지 않기

## 11. 기술 구현 명세

### 11.1 페이지 렌더링 전략

| 페이지 | 전략 | 이유 |
|--------|------|------|
| 허브 페이지 | SSG + ISR | 거의 안 변하지만 가끔 갱신 |
| 카테고리 페이지 | SSG + ISR (1일) | 새 오름 추가 시 갱신 |
| 개별 오름 페이지 | SSG + ISR (1주) | 후기·사진 자주 갱신 |
| 사용자 페이지 | SSR 또는 CSR | 동적 데이터 |

ISR (Incremental Static Regeneration): Next.js 기능. 백그라운드에서 페이지 재생성.

### 11.2 빌드 시간

360개 페이지 × 평균 1초 빌드 = 6분.
- Vercel 기본 빌드 시간 한도: 45분 (충분)
- 점진적으로 늘어날 시 ISR로 분산

### 11.3 이미지 최적화

- WebP/AVIF 자동 변환 (Next.js Image)
- Lazy loading 기본
- Cloudflare Images로 동적 리사이징

### 11.4 성능 목표

- LCP (Largest Contentful Paint): < 2.5초
- 페이지 크기: < 200KB (gzip)
- 첫 페이지 로드 (3G): < 5초

## 12. 리스크와 대응

### 12.1 AI 생성 콘텐츠 페널티

Google이 AI 생성 콘텐츠를 점점 감지·페널티 부과 추세.

**대응**:
- AI 생성 후 **반드시 인간 검수**
- 단순 AI 출력물 그대로 게시 금지
- 인간의 직접 관찰·경험 반드시 추가
- E-E-A-T (Experience, Expertise, Authoritativeness, Trustworthiness) 강화

### 12.2 사실 오류 (할루시네이션)

AI가 가짜 정보를 사실처럼 생성하는 문제.

**대응**:
- 검증 가능한 정보만 작성하는 정책
- DB 메타 데이터를 단일 진실 공급원으로
- 인간 검수자의 사실 확인 필수
- 사용자 신고 채널 운영

### 12.3 검색 알고리즘 변경

Google·네이버는 알고리즘을 자주 업데이트.

**대응**:
- 단일 기법(SEO 트릭)에 의존하지 않기
- 본질적 가치 (좋은 콘텐츠, 좋은 UX)에 집중
- 분기별 SEO 트렌드 점검
- 다양한 트래픽 채널 분산 (SNS, 직접 유입 등)

### 12.4 경쟁 사이트 등장

같은 콘셉트의 사이트가 등장할 수 있음.

**대응 (해자 구축)**:
- 사용자 후기·사진 자산 (선점 효과)
- 인쇄물 패스포트 (디지털만 따라하기 어려움)
- 제휴 상권 네트워크 (아날로그 자산)
- 도메인 권위 (1년 이상의 콘텐츠 축적)
